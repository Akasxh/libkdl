1 — The Problem

MLIR compiles one gpu.module to 3 GPU vendors. OffloadBinary carries N device images. But at runtime, the offload stack picks the first compatible image — no metadata vocabulary, no published measurement, no "best-compatible" mechanism. We fix all three.

1 The Gap

Since Aug 2025, MLIR compiles NVIDIA + AMD + Intel GPUs in one pipeline. The compile-time half is done. The runtime selection policy is not — #gpu.select_object resolves at compile time only.

Why This Matters

CERN ~80 build configs for heterogeneous GPU clusters (A100 + MI250X + CPU). A fat binary with runtime selection eliminates combinatorial explosion.
vLLM Separate NVIDIA/AMD codepaths per kernel. Runtime variant selection enables one serving binary for both vendors.
Cloud AWS p4/p5, Azure ND — GPU model unknown at container image build time. Ship one binary, resolve at launch.
The Status Quo Cost
  • N build pipelines per vendor instead of one fat binary
  • image:cuda / image:rocm / image:sycl — 3× storage & CI
  • PyTorch, JAX, ONNX-RT each reimplement their own GPU dispatch
  • No published latency data on the GPU dispatch stack

2 System Architecture

C1: Metadata Vocabulary (5 New Keys)

KeyTierExampleSemantics
min_sm MUST "75" Min CUDA SM; rejects if device < value
min_gfx MUST "gfx90a:cdna2" Min AMD GFX within ISA family
requires_features MUST "tensor_core_nv,bf16" Named tokens; rejects if absent
variant_priority MAY "10" Higher = preferred among compatible
variant_tag MAY "optimized" Label: generic / optimized / fallback

Backward-compatible: Missing keys = no constraint. Old runtimes ignore unknown keys. No format version bump.

Selection Policy: rank_by_priority
  • Reads variant_priority key from each binary's StringMap
  • Returns highest-priority binary that passes all min_sm / min_gfx / requires_features guards
  • Tied priorities: falls back to insertion order (deterministic)
  • Pluggable SelectionPolicyInterface — future strategies (ISA-score, power-cap) slot in without changing the attribute
Key Design Properties
  • Backward-compatible: Falls back to index-0 if no metadata keys present. Existing #gpu.select_object users unaffected.
  • Compile-time emission: Selection logic emitted as IR during embedBinary() — no runtime library dependency.
  • CPU FMV analogy: Same pattern as target_clones IFunc resolvers — compile-time emission of runtime selection, now for cross-vendor GPU binaries.
  • Zero incremental cost: After one-time global_ctors selection, hot path is a single pointer load.

~780 LOC to implement OffloadingLLVMTranslationAttrInterface — a standalone RFC, independent of the metadata vocabulary patch.

1
Module loading dominates. cuModuleLoadData is ~90% of total cold dispatch (36.0 µs of ~40 µs). Selection is noise-floor (<0.01%).
2
Not the bottleneck. At 4–6 ns, a scan of 3 entries is 7,000× cheaper than a single kernel launch. Scales to ~40 ns for 30 variants — still within noise.
3
Hot path: zero overhead. After one-time selection, the module handle is cached in a global pointer. Subsequent launches are identical to #gpu.select_object.
4
The ld.so analogy. Resolves in global_ctors, hot path is a single pointer load — same pattern as the dynamic linker's PLT, now for cross-vendor GPU binaries.
