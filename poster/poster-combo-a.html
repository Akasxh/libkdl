Bridging Runtime Gaps in LLVM:
Vendor-Agnostic Dispatch for ML Kernels

MLIR compiles one gpu.module to 3 GPU vendors. OffloadBinary carries N device images. But at runtime, the offload stack picks the first compatible image -- no metadata vocabulary, no published measurement, no "best-compatible" mechanism. We fix all three.

The Gap

Since Aug 2025, MLIR compiles NVIDIA + AMD + Intel GPUs in one pipeline. The compile-time half is done. The runtime selection policy is not -- #gpu.select_object resolves at compile time only.

cuBLAS, cuDNN, PyTorch, torch.compile all select GPU code at runtime. MLIR gpu.binary is the exception.

Upstream Evidence

Why This Matters

CERN ~80 build configs for heterogeneous GPU clusters (A100 + MI250X + CPU). A fat binary with runtime selection eliminates combinatorial explosion.
vLLM Separate NVIDIA/AMD codepaths per kernel. Runtime variant selection enables one serving binary for both vendors.
Cloud AWS p4/p5, Azure ND -- GPU model unknown at container image build time. Ship one binary, resolve at launch.

The status quo: N build pipelines per vendor, 3x storage & CI, every framework reimplements dispatch, no published latency data. We fix all of this.

System Architecture

MLIR gpu.module compiles to OffloadBinary with N variants. #gpu.runtime_select picks the best match at load time via metadata ranking. GPU launch path is unchanged.

KeyTierExampleSemantics
min_smMUST"75"Min CUDA SM; rejects if device < value
min_gfxMUST"gfx90a:cdna2"Min AMD GFX within ISA family
requires_featuresMUST"tensor_core,bf16"Named tokens; rejects if absent
variant_priorityMAY"10"Higher = preferred among compatible
variant_tagMAY"optimized"Label: generic / optimized / fallback

Backward-compatible: Missing keys = no constraint. Old runtimes ignore unknown keys.

C2: First Published Dispatch Flame Graph

GTX 1650 (sm_75), CUDA 13.1, null kernel CUBIN. cuModuleLoadData dominates at 36 us (90% of cold path). Selection overhead: 3-6 ns.

Selection Scales to 100 Variants

Linear O(n) scan stays negligible. Even at 100 variants, selection (62.5 ns) is 26,000x cheaper than module load.

Selection Overhead vs. Real Kernels

Selection overhead is sub-pixel at every kernel scale: 6x10^-6% for large GEMM to 6x10^-3% for micro-kernels.

Not the bottleneck. At 4-6 ns, a scan of 3 entries is 7,000x cheaper than a single kernel launch. Scales to ~40 ns for 30 variants.

Hot path: zero overhead. After one-time selection, the module handle is cached in a global pointer. Subsequent launches identical to #gpu.select_object.

The ld.so analogy. Resolves in global_ctors, hot path is a single pointer load -- same pattern as the dynamic linker's PLT, now for cross-vendor GPU binaries.
