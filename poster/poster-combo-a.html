In plain English: When you compile a GPU program for multiple vendors (NVIDIA, AMD, Intel), LLVM currently picks the first binary that works. We measure the cost and propose a way to pick the best one.

1 The Gap: Compile-Time Done, Runtime Missing

Tri-vendor compilation works. Since August 2025, MLIR's GPU dialect supports NVVM (NVIDIA Virtual Machine) + ROCDL (ROCm Device Library) + XeVM (Intel Xe GPU) targets in a single compilation pipeline.

  • gpu-module-to-binary produces a multi-vendor gpu.binary with N embedded device objects
  • #gpu.select_object resolves ONE target at compile time — index-based, no runtime path

Why This Matters Now

The pressure is real and growing:

CERN HEP-CCE

Maintains ~80 build configs for heterogeneous GPU clusters (A100 + MI250X + CPU). A fat binary with runtime selection eliminates combinatorial explosion.

vLLM

Separate NVIDIA/AMD codepaths per kernel. Runtime variant selection enables a single serving binary for both vendors.

Cloud Containers

AWS p4/p5, GCP A3 — GPU model is unknown at compile time for containerized ML workloads.

torch-mlir / ONNX-RT

Each framework reimplements dispatch for gpu.binary with 3 targets. Moving selection into LLVM eliminates per-framework duplication.

The Status Quo Cost

  • Each project maintains separate build pipelines per GPU vendor instead of a single fat binary
  • Container images must be duplicated: image:cuda, image:rocm, image:sycl — tripling storage and CI time
  • Selection logic is duplicated in every framework: PyTorch, JAX, TensorFlow, ONNX-RT each implement their own GPU dispatch
  • No published latency data on the GPU dispatch stack — developers optimize blindly without knowing where time is spent

2 C1: OffloadBinary Metadata Vocabulary

The OffloadBinary format (D122069, 2022) includes a flexible StringMap per image — but only 2 keys are standardized (triple, arch). Four years later, the vocabulary has not grown.

The consumer hook exists (PR #185663). The vocabulary does not. Until now.

5 New Standard Keys

KeyTierTypeExampleSemantics
min_smMUSTuint"75"Min CUDA compute capability (e.g. sm_75 = NVIDIA Turing, compute capability 7.5); rejects if device < value
min_gfxMUSTarch:family"gfx90a:cdna2"Min AMD GFX within ISA (Instruction Set Architecture) family (family-tagged)
requires_featuresMUSTcomma-list"tensor_core_nv,bf16"Named capability tokens; rejects if absent
variant_priorityMAYuint"10"Higher = preferred among compatible images
variant_tagMAYstring"optimized"Human label: generic, optimized, fallback, debug

Extension to isMetadataCompatible()

// Current (PR #185663): triple + arch only
// Extended (opt-in when keys present):
bool isMetadataCompatible(image, device) {
  if (image.Triple != device.Triple) return false;
  if (auto minSm = image.getString("min_sm"))
    if (parseSm(device.Arch) < stoul(*minSm))
      return false;
  if (auto feats = image.getString("requires_features"))
    for (auto &tok : split(*feats, ','))
      if (!device.hasCapability(tok)) return false;
  return true;
}

Backward-compatible: Missing keys = no constraint. Old runtimes ignore unknown keys. No format version bump, no ABI break.

launchKernel() loads from @kernels_module_ptr — identical codepath to SelectObjectAttr. Zero hot-path overhead after one-time selection.

Selection Policy: rank_by_priority

  • Reads variant_priority key from each binary's StringMap
  • Returns the binary with the highest numeric priority that passes all min_sm / min_gfx / requires_features guards
  • Tied priorities: falls back to insertion order (deterministic)
  • Policy is a pluggable SelectionPolicyInterface — future strategies (ISA-score, power-cap) slot in without changing the attribute

Key Design Properties

  • Backward-compatible: Falls back to index-0 if no metadata keys present. Existing #gpu.select_object users unaffected.
  • Compile-time emission: Selection logic emitted as IR during embedBinary() — no runtime library dependency, no dynamic linking overhead.
  • Strategy-pluggable: rank_by_priority uses variant_priority key; future strategies (e.g. ISA-score, power-cap) slot in via the attribute interface.
  • CPU FMV analogy: Same pattern as target_clones IFunc resolvers — compile-time emission of runtime selection, now for cross-vendor GPU binaries.

~780 LOC to implement OffloadingLLVMTranslationAttrInterface — a standalone RFC, independent of the metadata vocabulary patch.

Key Findings

1

Module loading dominates cold path. cuModuleLoadData is ~90% of total cold dispatch latency (36.0 µs of ~40.1 µs end-to-end). The selection algorithm is noise-floor (<0.01%).

2

Selection is not the bottleneck. At 4–6 ns, dispatch table scan for 3 entries is 7,000× cheaper than a single kernel launch (1.6 µs). Scales to ~40 ns for 30 variants — still within noise.

3

Hot path has zero overhead. After one-time selection, the module handle is cached in a global pointer. Subsequent launchKernel calls are identical to compile-time #gpu.select_object.

4

The ld.so analogy holds. Think of it like a phone app store that auto-installs the right version for your device. #gpu.runtime_select picks the right GPU binary once at program start; every subsequent kernel launch uses that choice with zero overhead. (Technically: resolves in global_ctors, hot path is a single pointer load — same pattern as the dynamic linker's PLT.)

Scope: Measured on GTX 1650 (sm_75). Layer fractions are hardware-specific; the decomposition methodology generalizes. AMD path validated via mocked HIP; physical ROCm pending.
