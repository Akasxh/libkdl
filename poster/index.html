The Problem

MLIR's gpu-module-to-binary pass compiles kernels to multiple GPU targets (NVPTX, AMDGCN, SPIR-V) simultaneously — but gpu.select_object resolves the target at compile time during LLVM IR translation. No upstream MLIR mechanism exists for runtime hardware detection to choose among compiled variants. The GPUOffloadingLLVMTranslationAttrInterface is the designed extensibility point, but no implementation performs runtime dispatch.

The multi-target binary infrastructure exists. The runtime dispatch layer does not.

Why Runtime Dispatch Matters

  • Mixed GPU clouds: AWS offers NVIDIA A100/H100, Azure provides AMD MI300X, GCP has TPUs + NVIDIA. Deploy one artifact everywhere.
  • cuBLAS runtime selection: Hundreds of GEMM kernels per precision; ML-trained recommender selects at runtime with 93% optimality.
  • cuDNN v9 on-the-fly compilation: Three heuristic modes + NVRTC-compiled fusion kernels based on compute capability at runtime.
  • PyTorch dispatcher: Computes DispatchKeySet on every operator call from live tensor metadata and thread-local state.
  • Helix (ASPLOS 2025): 3.3x throughput on mixed-GPU clusters via heterogeneous-aware scheduling.
  • vLLM piecewise CUDA Graphs: Runtime batch-size bucket selection. Heterogeneous GPU serving explicitly unsupported today.

"ML Kernels Are Static" Is Empirically False

  • cuBLAS: Hundreds of GEMM kernels; runtime ML recommender (93% optimal). cublasSetSmCountTarget() allows runtime SM count override.
  • cuDNN v9: NVRTC on-the-fly compilation based on compute capability.
  • torch.compile: Shape guards checked per invocation; changes trigger recompilation. Backend selection is manual.
  • CUDA Graphs: Each unique shape requires re-recording — multi-version dispatch in practice.
  • ExecuTorch 1.0: Runtime delegate selection for heterogeneous edge hardware.

Where dispatch adds value: heterogeneous serving (Helix: 3.3x), cloud portability, edge deployment on unknown hardware, multi-tenant inference with mixed GPU generations.

Survey: Compile-Time Portability

ALPAKA (Helmholtz/CERN)

Header-only C++20 library using Redundant Hierarchical Parallelism. 5-level abstraction (Grid/Block/Warp/Thread/Element). Backends: CUDA 12+, HIP 6+, SYCL, OpenMP, TBB. Performance: >94% of native CUDA. CMS Run 3 HLT production deployment. Limitation: backend fixed at compile time via CMake. No runtime dispatch.

Kokkos (Sandia/DOE)

Polymorphic memory layouts (Kokkos::View) + execution policies. P3 scores 0.82–0.99 — highest among all portability frameworks (Davis et al., ICS 2025). <5% overhead vs native. Limitation: compile-time backend selection only.

RAJA (LLNL)

Lambda-based execution policies. P3 scores 0.47–1.00, <3% overhead vs native CUDA. Best for low arithmetic intensity. Limitation: compile-time only, SYCL/Intel experimental.

HIP (AMD)

Thin header-swap: hip/hip_runtime.h resolves to CUDA or AMD headers. Zero runtime indirection on either path. ~400+ API functions, hipify-clang achieves ~90–95% automatic translation. Limitation: compile-time target, no tensor core portability, warp size differences (32 vs 64).

Survey: Runtime Portability Layers

SYCL (Khronos Standard)

Intel DPC++ (multi-pass fat binary) and AdaptiveCpp/hipSYCL (single-pass SSCP + runtime JIT). Runtime device selection via device_selector. P3 scores 0.46–0.65 — "appears to often perform worse than other models" (Davis et al.). On A100, ALPAKA and Kokkos matched native CUDA while SYCL was ~10x slower. SYCL-MLIR compiler (CGO 2024) achieved up to 4.3x speedup over DPC++.

AdaptiveCpp SSCP

Single-Source Single-Compiler Pass: generic LLVM IR in host binary, runtime JIT to PTX/SPIR-V/amdgcn. ~15% first-launch overhead, near-zero cached. >2x faster than multi-pass for 3 AMD architectures. Closest existing implementation to multi-versioned kernel dispatch.

OpenCL & Vulkan Compute

OpenCL: Source portability ≠ performance portability. 3.0 retreated to "1.2 mandatory, everything optional." Key legacy: SPIR-V, ICD dispatch. Vulkan: Broadest hardware coverage. ~70–80% of CUDA on NVIDIA; on AMD RDNA3, Vulkan matches or exceeds ROCm/HIP. Cooperative matrices can exceed CUDA in LLM scenarios. IREE achieves SOTA results on RDNA3 via Vulkan.

Survey: Compiler & JIT Systems

Triton (OpenAI)

Python DSL → 9 custom MLIR dialects → PTX/AMDGCN. Most MLIR-native system in the ML compiler space. Matches/exceeds cuBLAS for standard shapes. No runtime dispatch — backend chosen at compile time. NVIDIA CC 8.0+, AMD ROCm 6.2+ only.

TVM (Apache) & XLA (Google)

TVM: relax + TensorIR with MetaSchedule auto-tuning. Near-cuBLAS after hours of tuning; untuned 2–5x slower. Core team acquired by NVIDIA (OctoAI, 2024). XLA: StableHLO (MLIR dialect) → LLVM codegen. PJRT provides hardware-agnostic runtime API. Neither supports runtime dispatch.

Proteus (CGO 2025)

Lightweight portable JIT: embeds LLVM IR, recompiles at runtime with device-specific constant folding. Up to 2.8x speedup on AMD, 1.78x on NVIDIA vs AOT. Limitation: targets one vendor at a time. JIT overhead median = ~900 kernel executions.

OCCA (DOE/Shell)

OKL directive-annotated C compiled at runtime via JIT. 6.07s OCCA vs 8.02s RAJA on structured grid. Limitation: not standard C/C++, no MLIR integration, niche adoption.

SPIR-V as Universal GPU IR

SPIR-V achieves 50–80% of native performance across vendors (arXiv:2603.28793). Key limitations blocking universal adoption:

  • Structured CFG only — no irreducible control flow (required by some optimized kernels)
  • No async memory ops — cp.async (NVIDIA), buffer_load (AMD) have no SPIR-V equivalent
  • Cooperative matrix gap — VK_KHR_cooperative_matrix exists but vendor support varies wildly
  • Vendor-flavored SPIR-V — AMD accepts RDNA-specific extensions; Intel IGC is the only first-class SPIR-V consumer

GapEvidence

Gap 2: No lightweight standalone dispatch layerIREE = 100K+ LOC. Upstream MLIR ExecutionEngine fixes targets at JIT time. No ~500 LOC shim exists.
Gap 3: No cost-model-driven variant selectionIREE uses first-valid-match. CUDA fatbin selects best SM match. No cross-vendor cost-model selection.

Proposal: libkdl — Kernel Dynamic Linker

Like ld.so resolves shared libraries at runtime, libkdl resolves GPU kernels based on available hardware. ~5K LOC C, zero external dependencies.

Key API

/* Deploy one binary to any GPU */
kdl_ctx ctx;
kdl_init(&ctx);                     // dlopen libcuda/libhip
kdl_load_bundle(ctx, "model.mtb", &b);  // parse MTB
kdl_select_kernel(ctx, b, "matmul", -1, &k); // auto-select
kdl_launch(k, gx,gy,gz, bx,by,bz, smem, args);
kdl_finalize(ctx);

Cost Model & Overhead

Test = FLOPs / min(Peakcompute, BW × Iarith) + Tlaunch

Each variant embeds a compute profile (FLOPs, bytes, arithmetic intensity). libkdl queries device peak TFLOPS + bandwidth at init, estimates wall-clock time, picks lowest-cost (device, variant) pair.

Integration Paths

torch.compile: FX graph → torch-mlir → MLIR linalg → gpu-module-to-binary → MTB → kdl_select_kernel() at inference. No auto hardware detection exists today — libkdl fills this gap.

ONNX Runtime EP: KdlExecutionProvider claims ops → torch-mlir → MLIR compile → MTB → kdl dispatch per subgraph. Complementary to IREE (full stack).

Relationship to IREE: complementary. IREE = best for end-to-end deployment with full control (100K+ LOC). libkdl = best for adding multi-target dispatch to existing MLIR pipelines (~5K LOC).

Future Work

  • Upstream MLIR integration: Propose runtime-aware GPUOffloadingLLVMTranslationAttrInterface handler to MLIR upstream.
  • Online autotuning: Profile-guided cost model refinement during serving warm-up.
  • SPIR-V backend: Add Vulkan/Level Zero dispatch for Intel GPU support.
  • Production bundle format: Migrate MTB to llvm-offload-binary for ecosystem compatibility.
