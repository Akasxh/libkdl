1 The Problem

MLIR's gpu-module-to-binary can compile a single kernel to NVPTX, AMDGCN, and SPIR-V simultaneously -- but gpu.select_object resolves the target at compile time during LLVM IR translation, not at runtime.

No upstream MLIR mechanism performs runtime hardware detection to choose among compiled variants. The dispatch decision is baked into the binary.

Why it matters now

  • Mixed GPU clouds -- AWS (NVIDIA A100/H100), Azure (AMD MI300X), GCP (TPU + NVIDIA)
  • HPC clusters -- Frontier (AMD MI250X), Aurora (Intel), Perlmutter (NVIDIA A100)
  • Edge deployment -- ExecuTorch 1.0 targeting unknown hardware at install time
  • Multi-tenant inference -- mixed GPU generations sharing one serving binary

Helix (ASPLOS 2025): 3.3x throughput on mixed GPU clusters by exploiting heterogeneous hardware -- but requires manual kernel management.

2 "ML is Static" -- Empirically False

A common objection is that ML kernels are well-known at compile time. Production evidence shows pervasive runtime dispatch:

  • cuBLAS -- Hundreds of GEMM variants per precision. ML-trained recommender selects at runtime (93% of optimal).
  • cuDNN v9 -- Three heuristic modes + runtime fusion engines that NVRTC-compile kernels on-the-fly.
  • PyTorch dispatcher -- Computes DispatchKeySet on every operator call from live tensor metadata.
  • torch.compile -- Guards checked at every invocation; shape changes trigger recompilation. Backend selection is manual.
  • vLLM -- Piecewise CUDA Graphs with runtime batch-size bucket selection. Heterogeneous serving explicitly unsupported.
  • CUDA Graphs -- Each shape requires re-recording; practitioners maintain one graph per shape bucket.

JIT + autotuning achieves >230% improvement over vendor implementations for LLM kernels (arXiv:2505.03780). Static dispatch leaves performance on the table.

4 Our Contribution

mlir-hetero-dispatch / libkdl

A Kernel Dynamic Linker -- analogous to ld.so for shared libraries. Just as ld.so resolves symbols from the library search path at process start, libkdl resolves kernel binaries from available GPU hardware at kernel launch.

Six components

  • Multi-target AOT -- MLIR's gpu-module-to-binary compiles one kernel to NVPTX + AMDGCN + x86
  • Routing table -- maps (kernel, vendor, arch) to binary offset
  • Capability contracts -- each variant declares: requires {cuda >= 11.0, sm >= 80, smem >= 48KB}
  • Cost-model ranking -- roofline model estimates execution time per device
  • Fallback chain -- best GPU, then next, then CPU. Always runs.
  • MTB bundle format -- single file containing all variants + metadata

Prototype status: NVIDIA + CPU dispatch working, AMD path in progress. MLIR pass integration via GPUOffloadingLLVMTranslationAttrInterface is future work.

6 MTB Bundle Format

Single-file Multi-Target Bundle containing all compiled variants, capability contracts, and a routing table:

+==================================+ | Header: "KDL_MTB\0" + version | +----------------------------------+ | Kernel Table | | name -> [variant indices] | +----------------------------------+ | Variant Table | | target | arch | contract | | | binary_offset | binary_size | +----------------------------------+ | String Table (NUL-terminated) | +----------------------------------+ | Binary Blobs | | [cubin] [hsaco] [x86 ELF] | +==================================+

Capability Contract (JSON)

{ "target": "nvptx", "min_arch": "sm_80", "min_shared_mem_kb": 48, "compute_profile": { "flops": 2e9, "arithmetic_intensity": 250.0 } }

Relationship to IREE

Complementary, not competing. IREE is ideal for full model deployment with its VM and HAL (100K+ LOC). libkdl adds multi-target dispatch to existing MLIR pipelines in under 1000 LOC -- embeddable anywhere MLIR is already used.

9 Future Work
  • Upstream MLIR RFC -- runtime-aware GPUOffloadingLLVMTranslationAttrInterface handler for gpu.select_object
  • Auto-tuning integration -- profile-guided cost model refinement using runtime telemetry
  • SPIR-V backend -- Vulkan compute dispatch for Intel/mobile GPUs (0-50% faster than ROCm on RDNA3)
  • vLLM integration -- heterogeneous GPU serving with automatic kernel routing per device
  • Kernel cache -- persistent MTB cache with version-aware invalidation

Key extensibility point: MLIR's GPUOffloadingLLVMTranslation-
AttrInterface is the upstream hook for a runtime-aware handler -- our design targets this interface directly.
