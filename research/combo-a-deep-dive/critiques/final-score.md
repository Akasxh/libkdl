## Dimension Rationale

### 1. Novelty: 3.5/5
The gap is real: `OffloadBinary` carries only `triple`+`arch`, and `#gpu.select_object` locks at compile time. The IFunc analogy is apt. However: the metadata vocabulary (5 string keys) is incremental engineering, not a research contribution. The layer decomposition is "first published" but the methodology (null kernel timing via CUDA driver API) is standard GPU benchmarking. No conceptual surprise.

**+0.5 fix:** Add a second GPU (AMD via ROCm or even Intel Arc) to show cross-vendor selection actually working end-to-end, not just NVIDIA-only with mocked HIP. One real AMD measurement would transform the novelty claim from "we propose" to "we demonstrate."

### 2. Soundness: 4.0/5
Strong. 3-run pinned data with CV% reported per layer. CIs computed with correct t-distribution (t_{0.025,2}=4.303). Cold/warm paths clearly separated. The 46 us kdl vs 3 ns PoC distinction is explicitly called out (paper lines 220-231). Paper footnote correctly disclaims that cold path includes `cuInit`+`cuCtxCreate`. Arithmetic checks out: 35,953+63+1,650+2,454 = 40,120 ns, and 35,953/40,120 = 89.6%.

Deduction: only 3 runs. For a peer-reviewed venue this is bare minimum. The warm L2 p99 CV is 117.8% (driven by Run 1 outlier) -- the paper wisely uses medians but does not acknowledge this instability.

**+0.5 fix:** Run 5 pinned runs (not 3) to tighten CIs. Add a sentence acknowledging the warm L2 p99 variance: "Warm module-load tail latency exhibits high variance (p99 CV 118%) due to sporadic driver-internal GC; medians are stable (CV 1.2%)."

### 3. Significance: 3.0/5
The LLVM audience will find the layer decomposition interesting but actionable impact is limited. Five metadata keys are a good idea but trivial to implement (~130 LOC by the paper's own estimate). `#gpu.runtime_select` is design-only -- zero MLIR C++ exists. The paper is honest about this but it means the "mechanism" contribution is a sketch, not a validated artifact. The CERN 80-build-config motivating example is compelling but the paper doesn't show how this work actually reduces those configs.

**+0.5 fix:** Implement a minimal `RuntimeSelectAttr` that compiles and passes MLIR lit tests (even selecting between 2 NVIDIA variants). This converts C3 from "design sketch" to "working prototype." The 883-line `.cpp.sketch` already exists per context.md -- flesh it into compilable code.

### 4. Presentation: 4.0/5
4-page paper, cleanly formatted, no LaTeX errors. Three clear contributions labeled C1/C2/C3. MLIR code listings are accurate. Known design issues are self-reported (static init ordering, sanitizer interaction). The upstream path with LOC estimates is concrete. 28 Q&A cards and an elevator pitch show serious preparation.

Deduction: the TikZ latency chart (lines 236-253) is crude -- bar widths are hardcoded, not data-driven. Acceptable for a poster but not publication-quality.

**+0.5 fix:** Replace the TikZ chart with a proper pgfplots horizontal bar chart or a pre-rendered SVG. Add the hot-path 4.1 us as a separate color in the same figure for immediate cold-vs-hot visual comparison.

### 5. Reproducibility: 4.0/5
Consumer hardware (GTX 1650), all commands documented in pinned-benchmark-results.md Section 8, CPU pinning via `taskset`, exact driver/CUDA versions recorded. `bench_layers.c` and `runtime_select_poc.c` are self-contained C with only `libcuda.so.1` as runtime dependency.

Deduction: the null kernel CUBIN (`/tmp/null_sm75.cubin`) is not version-controlled or generated by a checked-in script. A reproducer needs to know how to create it. Also, `bench_dispatch` requires the full libkdl build (5100 LOC) which has build dependencies not documented in the benchmark file.

**+0.5 fix:** Add a `generate_null_cubin.sh` script that produces the test CUBIN from inline PTX (`nvcc --ptx` or `ptxas`). Check it into `experiments/prototype/src/`. Document libkdl build deps in a one-line `apt install` command.

## Summary

Solid engineering poster with honest claims and well-measured data. The main weakness is that all three contributions stay in "proposal/measurement" territory -- none cross into "landed upstream code." For EuroLLVM this is acceptable (posters are often early-stage), but the composite sits at 3.7, not 4.5. The single highest-leverage improvement is making `RuntimeSelectAttr` compile against MLIR trunk, which would boost both Novelty and Significance by 0.5 each, pushing composite above 4.0.
