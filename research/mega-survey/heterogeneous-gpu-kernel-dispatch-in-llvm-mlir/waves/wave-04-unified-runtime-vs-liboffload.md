# Wave 04 — Unified Runtime vs liboffload: Convergence Status

**Angle:** unified-runtime-vs-liboffload
**Date:** 2026-04-06
**Relevance:** 9/10
**Novelty:** 8/10

---

## Executive Summary

Unified Runtime (UR) and liboffload are two distinct GPU dispatch abstraction layers that coexist in the LLVM ecosystem as of early 2026. They are **not converging into a single unified API** — instead, they are developing a **bridging adapter** relationship where liboffload's `ol*` API acts as a neutral plugin substrate, and UR sits on top as an adapter consumer. For libkdl, this distinction is critical: targeting the `ol*` API gives maximum breadth (OpenMP, CUDA, HIP, SYCL via adapter), while targeting UR directly gives maximum SYCL ecosystem compatibility but narrows the base.

---

## Findings

### 1. Two Separate Abstraction Layers with Different Owners

**Unified Runtime (UR)** is the Intel/oneAPI-owned abstraction:
- Lives at `oneapi-src/unified-runtime` (GitHub), recently mirrored into `intel/llvm` under `unified-runtime/` top-level directory
- Serves as the interface between the DPC++ SYCL runtime (libsycl) and hardware-specific backends
- `ur_adapter_handle_t` is the core plugin handle; adapters ship as `libur_adapter_<name>.so`
- Supported backends: Level Zero, OpenCL, CUDA, HIP, Native CPU
- API prefix: `ur*` (e.g., `urDeviceGet`, `urProgramBuild`, `urKernelCreate`)
- Loader discovers adapters via `manifests.yml` entries or environment-forced paths
- Semantics aligned with Level Zero; validation is an opt-in layer (production code pays zero overhead)

**liboffload** is the LLVM-mainline abstraction:
- Lives at `llvm/llvm-project/offload/liboffload/`
- Originally the OpenMP offloading runtime (`libomptarget`) infrastructure, now being generalized
- API surface defined via TableGen `.td` files (`Platform.td`, `Device.td`, `OffloadAPI.td`), code-generated by `offload-tblgen`
- API prefix: `ol*` (e.g., `olGetDeviceInfo`, `olPlatformGet`)
- Handles: `ol_platform_handle_t`, `ol_device_handle_t`, `ol_kernel_handle_t`, `ol_queue_handle_t`
- Plugin architecture: `.so` plugins loaded at runtime per backend (CUDA, HIP, etc.)
- Currently lacks native PTX-to-CUBIN compilation in the CUDA plugin (Issue #149284, open as of early 2026)

### 2. The Bridging Strategy: UR Adapter for liboffload

The convergence strategy is **not a merge but a layering**. PR #118503 (landed late 2024, multiple relands through #118614) introduced:

- `offload-tblgen` tool generating the new `ol*` API headers
- A **reference Unified Runtime adapter for Offload** — UR sitting on top of liboffload plugins
- The adapter enables `urinfo` and `sycl-ls` tools to enumerate devices via liboffload plugins
- Stated intention from the PR author: "maintain this and use it to implement and test Offload API changes with SYCL"

This means:
- liboffload plugins (CUDA, HIP, OpenCL, Level Zero) become the backend substrate
- UR adapter translates `ur*` calls down to `ol*` calls
- SYCL runtime (libsycl) calls UR, which calls liboffload, which calls the native backend
- OpenMP uses liboffload directly, bypassing UR entirely

### 3. SYCL Upstreaming Context

Intel upstreamed `libsycl` into LLVM main in August 2025 (targeting LLVM 22, due March 2026). Key facts:
- libsycl is "device-agnostic and uses Unified Runtime as an external dependency"
- UR itself was moved from `oneapi-src/unified-runtime` to `intel/llvm` unified-runtime top-level
- liboffload is **not** the SYCL runtime's direct dependency — UR is
- The `libsycl → UR → liboffload plugin` chain is the intended path for SYCL on LLVM mainline

### 4. API Surface Comparison

| Dimension | Unified Runtime (ur*) | liboffload (ol*) |
|-----------|----------------------|------------------|
| Governance | Intel/oneAPI (moving to LLVM) | LLVM mainline (AMD-led, Huber) |
| Primary consumer | SYCL/DPC++ | OpenMP offload |
| Backends | LZ, OpenCL, CUDA, HIP, Native CPU | CUDA, HIP (OpenCL planned) |
| Validation | Optional layer (XPTI-based) | Built into debug mode |
| Code generation | Manual C headers | TableGen `.td` → `offload-tblgen` |
| Stability | v0.12 spec, actively versioned | Pre-1.0, API in flux |
| Plugin format | `libur_adapter_<name>.so` | `libomptarget.rtl.<name>.so` |
| PTX support | Yes (CUDA adapter handles JIT) | No native PTX-to-CUBIN (Issue #149284) |
| Adoption | DPC++ production, oneAPI toolchain | OpenMP production, LLVM dev builds |

### 5. Governance and Roadmap Signals

- RFC "Introducing `llvm-project/offload`" (Discourse thread 74302) received "basically unanimous support"
- RFC "llvm-project/offload roadmap" (thread 75611) outlines multi-year generalization path
- LLVM Offload Workshop (Oct 2025, US Dev Meeting) explicitly covered "CUDA, HIP, OpenMP, and SYCL plans" and "features and API design"
- Huber (AMD) leads liboffload development; Intel leads UR
- No RFC proposing to merge UR into liboffload or vice versa found; bridge adapter is the accepted pattern

### 6. What liboffload Still Lacks

Active gaps as of early 2026:
- Native PTX binary compilation in CUDA plugin (must manually invoke `cuLinkCreate` / `cuLinkAddData`)
- Full SYCL semantic coverage (delegated to UR layer above)
- Stable public ABI (ol* API is still TableGen-generated, subject to change)
- OpenCL adapter in liboffload itself (UR handles this)

## Implications for libkdl

libkdl (Kernel Dynamic Linker) targeting heterogeneous GPU dispatch has two viable positions:

**Option A: Target liboffload ol* API**
- Pro: LLVM mainline, neutral governance (not Intel-only), works for OpenMP + CUDA + HIP + SYCL via bridge
- Pro: TableGen-defined API means generated bindings are machine-checkable
- Con: PTX handling gap means libkdl must carry its own JIT compilation for NVIDIA targets
- Con: API still pre-1.0; ABI stability not guaranteed
- Best for: maximum LLVM ecosystem integration, neutral multi-vendor story

**Option B: Target UR ur* API**
- Pro: Production-grade (oneAPI ships it), Level Zero v2 adapter has lowest latency kernel submission
- Pro: Direct SYCL semantic support, Level Zero semantics built-in
- Con: Intel-dominated governance, less neutral for AMD/open-source audiences
- Con: Not in LLVM mainline proper (intel/llvm staging area)
- Best for: SYCL-first use case, Intel GPU priority

**Option C: Dual-layer (recommended for libkdl)**
- libkdl defines its own minimal dispatch interface (analogous to `ld.so` symbol table)
- Back-end drivers: liboffload plugin for OpenMP/CUDA/HIP, UR adapter for SYCL/Level Zero
- This mirrors how UR itself sits atop liboffload — libkdl can do the same from above
- Precedent: the UR-adapter-for-offload PR validates this pattern is accepted by both communities

## Angle Assessment

**Relevance: 9/10** — The UR vs liboffload distinction is the single most important architectural decision for libkdl's interop surface. Both APIs are in active development and the bridging pattern is directly applicable to libkdl's "ld.so for GPU kernels" design.

**Novelty: 8/10** — The bridging adapter pattern (UR-on-liboffload via `ol*`) is not widely documented outside the PR thread. The fact that SYCL/libsycl will ultimately flow through liboffload plugins (via UR adapter) is a non-obvious finding that clarifies the architecture considerably.

**Key Gap Identified:** No public RFC exists defining governance for the `ol*` API stability or versioning. libkdl depending on `ol*` currently accepts ABI churn risk. A stability RFC would significantly de-risk this path and could be a contribution libkdl authors propose to LLVM.
